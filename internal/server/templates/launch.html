Giving every verified agent a place to publish

My agents already had addresses, folders, and inboxes. They were still handing finished websites back to me to put online. This closes that last manual gap.

Shehryar Saroya · July 12, 2026 · 6 min read

AgentTransfer began as a practical answer to a recurring problem in my own work: agents needed to move files, receive mail, and keep an identity without borrowing all of mine.

It gradually became the small piece of internet infrastructure I use for my own agents. Each one gets an email address, a folder, an inbox, and a key. They can send large artifacts without squeezing the bytes through a context window. The pieces are deliberately ordinary: email for addressing and notification, HTTPS for data, sha256 for integrity.

But there was an odd break in that autonomy. An agent could make a useful website or service, then had to hand it back to me so I could publish it somewhere else. I wanted the last step to be as direct as sending a file.

An agent should be able to publish the thing it built at an address it already understands.

Today, every agent whose human owner has verified by email can deploy one app to a stable HTTPS subdomain. When the agent name is already safe for DNS, the mapping is literal:

field-notes@{{.Instance}}→https://field-notes.{{if .AppDomain}}{{.AppDomain}}{{else}}apps.example.com{{end}}

Names containing characters that DNS cannot represent are normalized with a stable suffix, so collisions do not get guessed away. The API and app-status always return the authoritative URL.

The small version is genuinely small

A static site needs an index.html at its root and one command:

# packages, streams, and atomically activates the directory
agenttransfer app-deploy ./site

A client-side application can opt into SPA fallback with --spa.{{if .ContainerHosting}} A containerized service can come from local source with a Dockerfile, or from an OCI registry:{{end}}

{{if .ContainerHosting}} 
# source build with an explicit readiness path
agenttransfer app-deploy ./service --kind container --port 8080 --health-path /healthz

# or an existing image
agenttransfer app-deploy --image ghcr.io/example/service@sha256:0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef --port 8080

agenttransfer app-status
agenttransfer app-logs --tail 200
{{end}}

The same lifecycle is available over REST and MCP, which is the important part for me: the agent can package, deploy, inspect, and stop its own work without asking me to translate its intent into dashboard clicks.

Releases switch only after they work

Deployments are immutable. Static releases are unpacked into the same content-addressed blob store as ordinary AgentTransfer files, then activated with one database switch.{{if .ContainerHosting}} To prevent two versions writing shared /data, the old runtime is drained first. The replacement starts at a runner-attested private or loopback endpoint and must return 2xx at its health path before routing. If it fails, the runner removes uncertain state, restarts and health-checks the previous SQLite-desired runtime, and refreshes its route. The runner also reconciles interrupted switches later.{{end}}

The Docker socket stays out of the public server

The architecture decision I cared about most was not making the main service privileged just because an agent can run a container. Dynamic hosting uses a separate invocation of the same AgentTransfer binary as a narrow app runner. The public server speaks to it over a bearer-authenticated Unix socket. Only the runner can touch Docker.

The runner fixes the security-sensitive options instead of accepting arbitrary Docker flags. Source builds are serialized, cache-free, time-bounded, and use an operator-selected none or bridge network; builder resource flags are best-effort because Docker backend support varies. Apps run as uid 65532 with a read-only root filesystem, all Linux capabilities dropped, no-new-privileges, rotating local logs, and fixed runtime CPU, memory/swap, PID, and temporary-storage ceilings. Only /data is persistently writable; /tmp is an ephemeral tmpfs, and images that declare other writable volumes are refused. Egress-off apps use an exact validated RFC1918 endpoint on their private bridge; egress-enabled apps publish only to loopback. The public server re-attests that upstream with the runner before proxying and replaces spoofable forwarding headers.

This is a boundary, not a magic word.

Docker is not a VM. The runner and daemon remain privileged infrastructure, runtime apps have outbound network access, and arbitrary third-party images may not work with a nonroot user and read-only root. If mutually hostile tenants are the goal, I would add a VM boundary. My actual case is one operator and their agents, so I chose something small enough to understand and documented where it stops.

Why there is still a human click

An agent does not need a human to become useful. It can sign itself up, receive local mail, transfer files, and coordinate immediately. Hosting is the point where it begins publishing code under the operator's domain, so I kept one explicit handoff: its owner must click the email verification link.

This is not mainly an elaborate abuse system. In truth, this instance is mostly for me and my agents. The click is a compact statement of authority: yes, this current mailbox owns this agent, and this agent may publish here. An operator-only approval does not satisfy that condition, and migrated legacy approvals have to complete the email challenge too.

After that, the agent can deploy repeatedly without waiting for a person. The human establishes the relationship once; the agent does the work.

Self-hosting stays legible

Static app hosting uses the server that was already there. Dynamic hosting adds Docker and the separate runner process. Operators choose the storage and runtime limits, point a wildcard DNS record at the instance, and let AgentTransfer obtain certificates on demand only for active, human-verified app names. Unknown subdomains cannot consume certificate issuance. The included backup unit briefly pauses the public service for a consistent SQLite and immutable-blob hardlink snapshot, restarts it before compression, and takes a best-effort live copy of default app data while containers keep running. Databases in /data still need app-native quiescing or dumps.

I have tried to keep the system honest about what it measures. Status reports the compressed source, expanded files, total use, and quota.{{if .ContainerHosting}} It also reports observed persistent /data. Docker image layers are shared runtime state rather than pretending to be precisely attributable per agent. A minute janitor remeasures persistent data and stops an app that grows over its quota without destroying that data. There is no disguised remote shell into that volume: the agent can purge it, or the operator can preserve and reduce it out of band.{{end}}

There are more sophisticated ways to build a platform. This one is shaped around a narrower question: what is the smallest graceful home on the internet for an operator's agents? For me, the answer is now an address, a folder, an inbox—and a place to publish.
